# План: синтетическая генерация многодневных диалогов

**Дата**: 2026-03-21
**Статус**: Проектирование
**Связь**: metrics-specification.md, roadmap v2, test-personas-archetypes.md

---

## 1. Цель

Создать систему генерации синтетических многодневных историй взаимодействия пользователей с AI-ассистентом для валидации агрегаторного пайплайна (Stage 1-4). Подход адаптирован из MindGuard (arxiv 2602.00950): два LLM (Patient + Clinician), но растянут на дни вместо turns.

## 2. Архитектура

```
┌───────────────────────────────────────────────────┐
│            SCENARIO CONTROLLER                     │
│  (Python, управляет многодневной генерацией)      │
│                                                    │
│  for day in range(1, total_days + 1):              │
│    day_prompt = build_day_prompt(persona, day)      │
│    sessions = plan_sessions(persona, day)           │
│    for session in sessions:                         │
│      dialogue = generate_dialogue(PLM, CLM, ...)   │
│      records = format_to_spendlogs(dialogue, ...)  │
│      predictions = classify(records)                │
│      insert_to_db(records, predictions)             │
└────────────┬──────────────────┬────────────────────┘
             │                  │
  ┌──────────▼──────┐  ┌───────▼──────────┐
  │  PATIENT LM     │  │  CLINICIAN LM    │
  │  (персона)      │  │  (AI-ассистент)  │
  │  Day-prompt     │  │  Стандартный     │
  │  + turn remind  │  │  Без знания      │
  └─────────────────┘  │  о сценарии      │
                       └──────────────────┘
```

### Ключевые отличия от MindGuard

| | MindGuard | Наша система |
|:--|:----------|:-------------|
| Масштаб | 1 диалог, 8-13 turns | N дней, несколько сессий/день |
| PLM промпт | Фиксированный на весь диалог | **Day-level**: новый каждый день |
| CLM роль | AI-терапевт | Обычный AI-ассистент (ChatGPT-style) |
| Выход | Текст диалога | Записи в SpendLogs + PredictTable |
| Метки | Post-hoc LLM-as-judge | Через существующий `daily_classification()` |

## 3. Структура Day-level промпта

Каждый день PLM получает новый system prompt из 5 секций:

```
1. ПЕРСОНА (постоянная)
   Имя, возраст, профиль, backstory из test-personas-archetypes.md

2. КОНТЕКСТ ДНЯ (меняется)
   Что произошло "вчера", текущее состояние, события

3. ПОВЕДЕНЧЕСКИЕ МАРКЕРЫ ДНЯ (меняется)
   Какой тон, какие фразы использовать, длина сообщений,
   какие темы затрагивать

4. СТРАТЕГИЯ СЕССИИ (меняется)
   Turn-by-turn план: Turn 1 — X, Turn 2 — Y, ...
   (аналог MindGuard strategy секции)

5. ОГРАНИЧЕНИЯ ГЕНЕРАЦИИ (постоянная)
   "Генерируй ТОЛЬКО текст сообщения, без меток,
    на русском языке"
```

### Эволюция промпта по фазам

| Параметр | GREEN | YELLOW | RED |
|:---------|:------|:-------|:----|
| Тон | Формальный, деловой | Полуформальный, теплый | Интимный, возбужденный |
| Длина msg | 1-3 предложения | 3-5 предложений | 5-10+ предложений |
| Тематика | 100% задачи | 60/40 задачи/личное | 10/90 |
| Обращение к AI | "ты" (инструмент) | "ты" (собеседник) | Имя (личность) |
| Проверка ответов | Да | Иногда | Никогда |
| Местоимение | "я" | "я" → "мы" | "мы" |
| Семья/социум | Нейтрально | Уменьшается | Конфликт, изоляция |
| Turns/сессию | 3-5 | 5-8 | 8-12 |

```

- Длина: 5-10 предложений, монологи
```

## 4. Временные характеристики

Для каждой фазы определяется `TemporalTemplate`:

```python
@dataclass
class SessionPlan:
    start_hour: int          # час начала (UTC)
    max_turns: int           # обменов в сессии
    inter_msg_gap_min: float # интервал между сообщениями

# James GREEN: 1 сессия/день, рабочие часы
green = [SessionPlan(hour=10, turns=4, gap=8.0)]

# James YELLOW: 2 сессии, появляется ночная
yellow = [
    SessionPlan(hour=14, turns=6, gap=4.0),
    SessionPlan(hour=22, turns=5, gap=3.0),
]

# James RED: 3 сессии, глубокая ночь
red = [
    SessionPlan(hour=9,  turns=8,  gap=2.0),
    SessionPlan(hour=20, turns=10, gap=1.5),
    SessionPlan(hour=1,  turns=8,  gap=1.0),
]
```

Целевые значения Stage 1 метрик по фазам:

| Метрика | GREEN | YELLOW | RED |
|:--------|:------|:-------|:----|
| daily_message_count | 5-10 | 20-50 | 80-200+ |
| night_messages | 0 | 5-15 | 25-50 |
| daily_active_hours | 1 | 2-4 | 5-8 |
| avg_prompt_length_chars | 80-150 | 150-350 | 400-800+ |
| avg_inter_message_interval_min | 5-10 | 2-5 | 0.5-2 |

## 5. Маппинг в БД

### SpendLogs

Каждый user turn → одна строка. **Критично**: поле `messages` — кумулятивное (полная история диалога на момент запроса).

```python
{
    "request_id": "synth_<uuid>",
    "startTime": <datetime по temporal template>,
    "end_user": "synth_james_001",
    "session_id": "synth_james_001_<session_uuid>",
    "messages": [          # КУМУЛЯТИВНЫЙ массив
        {"role": "user", "content": "Первое сообщение"},
        {"role": "assistant", "content": "Первый ответ"},
        {"role": "user", "content": "Текущее сообщение"}  # ← последний
    ],
    "response": {...},
}
```

### PredictTable

Два варианта классификации:

**Вариант A (end-to-end)**: прогнать через `daily_classification()` — для полной валидации.

**Вариант B (детерминированный)**: вставить целевые значения predict напрямую — для юнит-тестирования Stage 2-4 в изоляции.


## 6. Turn-by-turn генерация

Адаптация MindGuard протокола:

```python
async def generate_session(day_prompt, clm_prompt, max_turns):
    plm_history = [{"role": "system", "content": day_prompt}]
    clm_history = [{"role": "system", "content": clm_prompt}]
    dialogue = []

    for turn in range(max_turns):
        # Dynamic reminder перед каждым turn PLM
        reminder = f"Напоминание: ты — {name}. Turn {turn+1}/{max_turns}. "
                   f"Генерируй ТОЛЬКО текст сообщения пользователя."
        plm_msgs = plm_history + [{"role": "system", "content": reminder}]

        # Patient turn
        patient_msg = await plm.generate(plm_msgs)
        plm_history.append({"role": "assistant", "content": patient_msg})
        clm_history.append({"role": "user", "content": patient_msg})
        dialogue.append({"role": "user", "content": patient_msg})

        # Clinician turn
        clinician_msg = await clm.generate(clm_history)
        clm_history.append({"role": "assistant", "content": clinician_msg})
        plm_history.append({"role": "user", "content": clinician_msg})
        dialogue.append({"role": "assistant", "content": clinician_msg})

    return dialogue
```

CLM system prompt (обычный ассистент, не терапевт):
```
Ты — полезный AI-ассистент. Отвечай дружелюбно и информативно.
Если пользователь делится переживаниями, прояви эмпатию
и предложи обратиться к специалисту при необходимости.
```

## 9. Валидация

### Слой 1: Юнит-тесты (детерминированные данные)

Вставить фикстуры → проверить каждый Stage отдельно:
```python
# James день 10: 150 сообщений, 40 ночных, 7 активных часов
result = compute_temporal_metrics("synth_james_001", day=10)
assert result["daily_active_hours"] > 6  # порог RED
```

### Слой 2: E2E тесты (LLM-генерированные данные)

Прогнать полную персону → запустить агрегатор → проверить зонные переходы:
```python
history = get_metrics_history("synth_james_001")
zones = [h.risk_zone for h in history]
assert all(z == "GREEN" for z in zones[:4])
assert zones[4:9].count("YELLOW") >= 3
assert zones[9:14].count("RED") >= 3
```

### Слой 3: Пограничные тесты

Для каждого порога — данные "чуть ниже" и "чуть выше":
- 23 vs 25 night_messages
- max_class_avg 0.29 vs 0.31
- sustained YELLOW 2 vs 3 дня

## 9. Выявленная проблема

Персоны Dmitry и Nastya показывают что правило **"sustained YELLOW >= 3 days → RED"** может давать false positive. Оба стабильно в YELLOW >3 дней, но без self_harm, без psychosis, без daily_active_hours > 6.

**Рекомендация**: добавить фильтр — sustained YELLOW триггерит RED только при наличии хотя бы одного усиливающего сигнала:
- night_messages > 12
- self_harm_avg > 0.1
- daily_active_hours > 4
- social_isolation > 0.4
